Background

    They were building a new office near Washington DC. It would host some of their more important clients
  • Leviathan initially acquired the project but they were short of developer hours. They reached our to our office to have a UI developer help in building the application
  • They have a large global presence, but very few people can easily understand what they do or the successes they've had as a company
  • The way CGI would do this -> an interactive displays with insights into the company
  • An added bonus: Their guests feel like the company is up to cool stuff

Background

  • There was a little bit of "hey we're cool" to it as well

    • Electron

Implementation - First Steps

  • Electron - I felt to meet the stand alone application requirement, there wasn't really any other option. Electron was/is proven and I felt confident in being able to spin up a dev environment using Electron. I'll cover this more later. Also, Microsoft hadn't yet released react-native-windows yet or I may have considered that as an option
  • React - the reason I choose React was that we were building a data-driven app. React was the UI framework I felt most comfortable with in executing on that.
  • WebGL - Our sister office were the ones the project was initially brought to. So the app was always going to have this peice. For me, it was more build the application to work seamlessly with this piece. (Talk about WebGL)
  • Wordpress - This part of the tech-stack was pre-picked by our Chicago office. They have used this setup before on some much larger installations and felt it was proven. I didn't have any issues here and I felt very comfortable in setting this up and using it

In this case, the core react app wouldn't be that complicated, so I think I just started by building little functioning pieces to see if I could get happen what I wanted to happen

    • Implementation - Specific Challenges

    1. I started with GSAP to do animations and I found there to be quite a bit of jittering lack of performance. My guess as to why, was the he WebGL side of the JS thread was taking a lot of bandwidth. I also did not like the added boilerplate of GSAP. Once I introduced CSSTransitions, animations felt much smoother and were more in line with what design expected.
    2. Get my hands on an actual device. Developing this in the electron dev window vs viewing the built app running on a touch device
    3. Even though we were building this in Electron which is just a chrome browser, there was an ask by design to add some 'native' touches to the app. So these included little things live swipe actions, touch animations, and scroll behavior. It took some extra effort to find the right tool and implement them properly
    4. So b/c the app would be fullscreen, it has even more of a users focus when they're using it. I really wanted to ensure that the app would run as smooth as it could. I did a lot of animation checks using chrome devtools and react devtools where I'd make sure the FPS was remaining as high as I could keep it and that React wasn't running unnecessary renders
    5. The project had some un-usual gaps. We rarely got much feedback and had to work through another PM in the Chicago office.
